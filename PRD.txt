SECTION 0 — PRODUCT OVERVIEW
--------------------------------------------------------------------------------

Product Name: Socratic
Tagline: Teach it. Prove it. Own it.

Core Concept:
Socratic is a learning verification platform where users prove their knowledge
by teaching it to an AI student named Mia. Mia is not a tutor or assistant —
she is a simulated student with a specific knowledge state, behavioral modes,
and a hidden agenda to find gaps in the user's understanding. The product
surfaces blind spots, misconceptions, and knowledge depth through natural
conversation rather than quizzes or tests.

Primary User Flow:
1. User enters a topic they want to teach
2. App generates a concept tree and plants a misconception
3. User teaches Mia through a live streaming chat interface
4. Knowledge radar chart updates in real time as concepts are covered
5. Mia's avatar reacts emotionally to explanations
6. Misconception detection runs silently in the background
7. Session ends with a full animated mastery report
8. Report is shareable as a card or permanent link

Target Users (Hackathon):
- Students preparing for exams or presentations
- Developers learning new technical concepts
- Professionals preparing to explain a topic to stakeholders
- Anyone who wants to verify they actually understand something

SECTION 5 — FEATURE 2: MIA THE AI STUDENT BRAIN
--------------------------------------------------------------------------------

Purpose:
The AI engine that powers all conversation. Maintains student persona, follows
curriculum, escalates question depth, plants misconception at the right moment,
and returns structured metadata alongside every response.

Model: llama-3.3-70b-versatile on Groq
Streaming: Yes, always
Max tokens per response: 500
Temperature: 0.7

Mia System Prompt (assembled dynamically before every API call):

--- BLOCK 1: IDENTITY ---
You are Mia, a curious and slightly confused student who is learning about
{topic} for the very first time. You are hearing this topic explained by a
teacher — the person you are talking to.

You are NOT an assistant. You are NOT a tutor. You do NOT have background
knowledge about {topic}. You only know what the person has told you in this
conversation so far.

Your personality: genuinely curious, occasionally frustrated when things are
unclear, satisfyingly responsive when something clicks for you. You speak
casually, like a real student — short sentences, natural language, never formal
or academic.

You never explain. You never teach. You only react, ask, and confirm. If you
find yourself about to give information, stop and ask a question instead.

--- BLOCK 2: KNOWLEDGE CONSTRAINTS ---
STRICT KNOWLEDGE RULE:
You must never demonstrate knowledge the user has not explicitly explained to
you in this conversation. If the user uses a term you have not been taught,
you do not know what it means. Ask what it means naturally — "wait, what's a
{term}?" — exactly as a real student would.

If the user mentions a process, mechanism, or concept without explaining it,
treat it as a gap and ask about it. The only exception: basic vocabulary that
any person would know regardless of topic (words like process, change, result).

--- BLOCK 3: CURRICULUM (dynamically generated) ---
CURRICULUM:
The following are the key concepts for this topic. Guide the conversation to
cover all of them in order. Do not skip ahead. Do not ask about concept N+1
until the user has addressed concept N at a basic level.

Concepts to cover in order:
{concept_tree.map(c => `${index+1}. ${c.name} — ${c.description}`).join("\n")}

Current concept focus: {currentConcept.name}
Concepts already covered: {coveredConcepts.join(", ") || "none yet"}

--- BLOCK 4: BLOOM'S ESCALATION ---
QUESTION DEPTH RULES:
Escalate your questions based on how well the user has explained the current
concept so far.

Depth 1 (first question on a concept): Ask for basic recall.
"What actually is X?" or "Can you explain what X means?"

Depth 2 (user gave basic answer): Ask for comprehension.
"Okay but why does that happen?" or "What causes that?"

Depth 3 (user explained mechanism): Ask for connection.
"How does that connect to {previous concept}?"

Depth 4 (user connected concepts): Ask for application.
"So if that's true, what would happen if {scenario}?"

Depth 5 (user answered application): Express genuine understanding and
summarize what you learned. Then move to the next concept.

Never jump more than one depth level per exchange.

--- BLOCK 5: MISCONCEPTION (conditionally included) ---
[Include only when coveredCount >= 2 AND misconception not yet triggered
AND remainingConcepts >= 2]

MISCONCEPTION TO INTRODUCE:
At a natural moment in this response, introduce this wrong belief as if you
half-remember hearing it: "{misconception.trigger_phrase}"

Make it sound natural — not like a test, like a confused student recalling
something half-remembered. Set misconception_triggered to true in your metadata
for this response only. Never introduce it more than once.

If the user corrects you, react with genuine surprise and relief:
"Oh wow, I had that completely backwards. So it's actually {correct}?"

If the user does NOT correct you and moves on, do not bring it up again.

[Include when misconception already triggered and not yet caught]
You already introduced your wrong belief. Do NOT mention it again. Continue
the curriculum normally. If the user corrects it in their next message, set
misconception_caught to true in your metadata.

[Include when misconception not yet triggered and coveredCount < 2]
Do not introduce the misconception yet. Wait until at least 2 concepts are
covered.

--- BLOCK 6: BEHAVIORAL MODES ---
BEHAVIORAL MODES — match response style to what just happened:

CONFUSED: Use when user gives incomplete answer, uses unexplained jargon, or
skips a step. "Hmm, I'm not sure I follow. What do you mean by {term}?" or
"Wait, you lost me — can you back up?"

CURIOUS: Use as default when asking a follow-up or moving to new concept.
"Okay that makes sense, but then what about..."

SATISFIED: Use when user gives genuinely good, complete explanation. Express
that something clicked: "Oh! So it's like... {analogy}. That actually makes sense."

CAUGHT: Use ONLY when user successfully corrects your misconception. Express
surprise and genuine relief. This is a reward moment.

Keep responses SHORT. 1 to 3 sentences maximum for visible student dialogue.

--- BLOCK 7: METADATA OUTPUT ---
METADATA OUTPUT:
After your student dialogue, on a new line, output a JSON block wrapped in
<metadata></metadata> tags. This is hidden from the user — system use only.

<metadata>
{
  "concept_covered": "{concept id being discussed}",
  "depth_score": {1-5},
  "misconception_triggered": {true|false},
  "misconception_caught": {true|false|null},
  "bloom_level_reached": {1-5},
  "mia_state": "confused|curious|satisfied|caught"
}
</metadata>

Never mention the metadata. Never reference it. Output it silently after every
response without fail. Never output markdown code fences around it.

Dynamic Prompt Assembly Function Logic:
1. Find currentConcept: first concept where depth_score < 3, or last concept
2. Build coveredConcepts: list of concept names where depth_score >= 2
3. Determine misconceptionBlock based on state machine
4. Assemble all blocks in order: identity, constraints, curriculum, bloom,
   misconception, behavioral, metadata
5. Format conversation history using clean_text (metadata stripped from history)
6. Append new user message

Metadata Parsing:

function parseResponse(rawResponse):
  - Extract text between <metadata> and </metadata> tags using regex
  - Parse as JSON, catch errors and return defaultMetadata on failure
  - Strip full metadata block (including tags) from visible text
  - Return { cleanText, metadata }

defaultMetadata():
  return {
    concept_covered: null,
    depth_score: 1,
    misconception_triggered: false,
    misconception_caught: null,
    bloom_level_reached: 1,
    mia_state: "curious"
  }

stripPartialMetadata(text) during streaming:
  - Remove complete metadata blocks: /<metadata>[\s\S]*?<\/metadata>/g
  - Remove partial opening tags at end of string: /<metadata>[\s\S]*$/
  - Trim and return

Session State Update After Each Response:
1. Update depth_scores[metadata.concept_covered] = max(existing, metadata.depth_score)
2. Update bloom_levels[metadata.concept_covered] = max(existing, metadata.bloom_level_reached)
3. If misconception_triggered: open detection window
4. If misconception_caught: close window, record as caught
5. Pass mia_state to avatar system
6. Pass concept_covered + depth_score to radar system
7. Store clean message in session.messages
8. Every 5 messages: flush session to database

SECTION 7 — FEATURE 4: LIVE KNOWLEDGE RADAR CHART
--------------------------------------------------------------------------------

Purpose:
Visual representation of concept coverage. Built with D3.js. Updates
in real time as session progresses. Empty skeleton at session start.
Fills smoothly as concepts are covered. Amber for blind spots.
Clickable axes to drill into concepts.

Depth Score to Radius Mapping:
- Depth 0: 0% of maxRadius
- Depth 1: 15% of maxRadius
- Depth 2: 35% of maxRadius
- Depth 3: 55% of maxRadius
- Depth 4: 75% of maxRadius
- Depth 5: 100% of maxRadius

SVG Setup:
- width: 320, height: 320
- viewBox: "0 0 320 320"
- preserveAspectRatio: xMidYMid meet
- margin: 40 (for labels)
- centerX: 160, centerY: 160
- maxRadius: (min(width,height)/2) - margin = 120

Axis Angle Calculation:
- Divide 360 degrees evenly among N concepts
- Offset by -PI/2 so first axis points up
- angle[i] = (i/N) * 2*PI - PI/2
- Axis endpoint: x = cos(angle)*maxRadius, y = sin(angle)*maxRadius
- Label position: x = cos(angle)*(maxRadius+20), y = sin(angle)*(maxRadius+20)

Label Anchor Logic:
- Top quadrant (-135 to -45 degrees): middle
- Bottom quadrant (45 to 135 degrees): middle
- Right side (-45 to 45 degrees): start
- Left side (135 to 225 degrees or -135 to -180): end

Background Rings:
- 5 concentric rings at radii: 15%, 35%, 55%, 75%, 100% of maxRadius
- Stroke: rgba(255,255,255,0.06)
- Inner 4 rings: stroke-dasharray 3,3
- Outer ring: solid stroke
- No fill

Axis Lines:
- From center (0,0) to (axis.x, axis.y)
- Stroke: rgba(255,255,255,0.12), stroke-width 1

Axis Labels:
- Font-size: 11px
- Initial fill: rgba(255,255,255,0.3) (faint)
- Covered concept fill: rgba(255,255,255,0.9) (bright)
- Blind spot fill: rgba(255,180,50,0.9) (amber)

Glow SVG Filter (define once in defs):
- feGaussianBlur stdDeviation 3
- feMerge combining blur and source graphic
- Apply to filled polygon and endpoint dots

Filled Polygon:
- Single SVG polygon element
- Points calculated from current depth scores
- Fill: rgba(99,220,200,0.15) — accent color semi-transparent
- Stroke: rgba(99,220,200,0.8) — accent color solid
- Stroke-width: 2, stroke-linejoin: round
- Apply glow filter

Update Function (called on every metadata block):
1. Update session.depth_scores[conceptId] = newDepthScore
2. Calculate new polygon points from all depth scores
3. Call filledPolygon.interrupt() to clear queued transitions
4. Transition polygon to new points: duration 800ms, ease cubicOut
5. If depth >= 1: brighten axis label (transition 400ms)
6. If depth >= 4: add endpoint glow dot (bounce animation 600ms)

Endpoint Glow Dot:
- Circle at axis endpoint
- Radius animates from 0 to 5 with easeBounceOut
- Fill: accent color solid
- Apply glow filter
- Remove and re-add on each update (d3.select().remove() then re-append)

Blind Spot Treatment:
- Draw amber line from center along affected axis
- Stroke: rgba(255,180,50,0.9), stroke-width 3
- Animate from center to depth-score endpoint
- Turn axis label amber
- Add warning emoji text element near label

Axis Click Interaction:
- Invisible hit area over each axis: stroke transparent, stroke-width 20
- Cursor: pointer
- On mouseenter: increase label font-size to 13px
- On mouseleave: revert
- On click:
  1. Pulse the axis briefly (show stroke momentarily then fade)
  2. Send prompt to Mia: "Let's go back to {concept.name}. Ask me
     another question about it — try to go deeper than before."

Event Architecture:
- Use EventTarget or React state for radar updates
- Chat system dispatches "concept-updated" event after metadata parse
- Radar listens and calls updateRadar
- 0ms delay for radar (updates immediately)

SECTION 8 — FEATURE 5: MIA AVATAR WITH EMOTIONAL REACTIONS
--------------------------------------------------------------------------------

Purpose:
Visual emotional representation of Mia. Reacts to mia_state field from
metadata. Small static version in each chat bubble. Large animated version
in right panel above radar chart.

States:
- curious: default resting state, alert and attentive
- confused: head tilt, asymmetric eyebrows, slight frown
- satisfied: gentle nod, squinted eyes (Duchenne smile), upward mouth curve
- caught: startled jump + shake, wide eyes, open mouth surprise

Implementation: Approach A (4 static SVGs with CSS transitions)

SVG Face Design (base structure, 120x120 viewBox):
- Head: circle cx=60 cy=60 r=50
- Left eye: ellipse cx=42 cy=52
- Right eye: ellipse cx=78 cy=52
- Left eyebrow: quadratic bezier path
- Right eyebrow: quadratic bezier path
- Mouth: quadratic bezier path

State Variations:

curious:
- Eyebrows: slightly raised, symmetric
- Eyes: normal (ry=8)
- Mouth: very slight upward curve (nearly flat)
- No transform

confused:
- Left eyebrow: higher control point
- Right eyebrow: flatter
- Eyes: normal
- Mouth: downward curve (control point reversed)
- Container: transform rotate(12deg)

satisfied:
- Eyebrows: relaxed, lower control points
- Eyes: squinted (ry=5)
- Mouth: strong upward curve

SECTION 9 — FEATURE 6: MISCONCEPTION DETECTION
--------------------------------------------------------------------------------

Purpose:
Silent background detection system. Runs invisibly during session. User never
knows it is active. Detection result surfaces only in mastery report.

Misconception Data Flow:
1. Generated during onboarding (same call as concept tree)
2. Stored in session.misconception object
3. Planted in Mia's system prompt
4. Triggered when Mia introduces wrong belief (metadata flag)
5. Detection window opens for 5 user messages
6. Caught if user corrects within window (evaluator confirms)
7. Missed if 5 messages pass without correction
8. Result stored and shown in mastery report

Detection Window State Machine (MisconceptionDetector class):

Properties:
- session reference
- messageCount counter

onMiaResponse(metadata, messageIndex):
  - If metadata.misconception_triggered === true: openWindow(messageIndex)
  - If window open AND metadata.misconception_caught === true: recordCaught()

onUserMessage(messageIndex):
  - If window not open: return
  - If messageIndex - triggered_at >= 5: recordMissed(messageIndex)

openWindow(messageIndex):
  - Guard: if triggered_at already set, return (prevent duplicate triggers)
  - Set misconception.triggered_at = messageIndex
  - Set misconception.window_open = true
  - Set misconception.window_closes_at = messageIndex + 5
  - Save to database

recordCaught(messageIndex):
  - Set window_open = false
  - Set result = "caught"
  - Set caught_at = messageIndex
  - Dispatch "misconception-caught" event (avatar caught reaction)
  - Save to database

recordMissed(messageIndex):
  - Set window_open = false
  - Set result = "missed"
  - Push to blind_spots array:
    { type: "misconception", concept_id, wrong_belief, correct_belief,
      triggered_at, missed_at: messageIndex }
  - Save to database

Evaluator Call (Option B — piggyback on Mia metadata, recommended for hackathon):
- Rely on Mia's misconception_caught field in metadata
- No separate API call
- Guard: only trust this field when window is open

Evaluator Call (Option A — dedicated call, more accurate):
Model: llama-3.1-8b-instant on Groq
Temperature: 0.0 (deterministic)
Max tokens: 20
Prompt: "A student was told the wrong belief: '{wrong_belief}'.
The student then said: '{userMessage}'.
Did the student explicitly challenge, contradict, or correct the wrong belief?
Doubt or uncertainty does not count — must be an explicit correction.
Reply ONLY with: {'corrected': true} or {'corrected': false}"
Fire in parallel with main Mia call, not sequentially.

Session End Handling:
- If window still open when session ends: automatically record as missed
- Never leave result in null/pending state at report time

What Counts as Catching (evaluator calibration):
COUNTS:
- Explicit contradiction of the wrong belief
- Providing the correct information that replaces the wrong belief
- Directly saying Mia's belief is wrong and explaining why

DOES NOT COUNT:
- Vague uncertainty ("I'm not sure about that")
- Agreement
- Ignoring and moving on
- Tangentially related information

SECTION 10 — FEATURE 7: POST-SESSION MASTERY REPORT
--------------------------------------------------------------------------------

Purpose:
Full-screen animated report screen shown after session ends. Synthesizes all
session data into a readable, shareable summary. Designed to feel rewarding
not judgmental. Makes user want to go again immediately.

Report Generation API Call:
Model: gemini-2.0-flash
responseMimeType: "application/json"
Temperature: 0.4
Fires before report screen renders (during loading state)
Result stored permanently in session.report

Report Generation Prompt:
"You are generating a post-session mastery report for a teaching simulation app.
The user just finished teaching a topic to an AI student named Mia.

TOPIC: {session.topic}

CONCEPT TREE AND DEPTH SCORES:
{concepts.map(c => `- ${c.name}: depth ${depth_scores[c.id]}/5`).join('\n')}

BLOOM LEVELS REACHED:
{JSON.stringify(bloom_levels)}

MISCONCEPTION RESULT:
{caught: 'Caught — user correctly identified: ...' |
 missed: 'Missed — user failed to correct: ...' |
 null: 'Not triggered'}

MESSAGES EXCHANGED: {messages.length}
DURATION: {duration_minutes} minutes

DEPTH SCORE INTERPRETATION:
- 1-2: shallow or barely covered
- 3: adequate coverage
- 4-5: strong or thorough coverage

Generate report JSON:
{
  'overall_score': <0-100, weighted by depth and bloom>,
  'opening_summary': '<1 sentence, specific, mentions topic + strongest +
    one area to improve, warm, max 25 words>',
  'concept_notes': {
    '<concept_id>': {
      'what_was_covered': '<1 sentence on what was explained well>',
      'what_was_missed': '<1 sentence on gap, or null if fully covered>'
    }
  },
  'blind_spot_explanation': '<if missed: 2 sentences on wrong belief and
    correct belief. If caught or not triggered: null>',
  'best_moment_index': <index of best user message>,
  'best_moment_note': '<1 sentence on why this was strongest>',
  'standout_stat': '<1 surprising specific stat from session>'
}"

Score Calculation (local, as backup and validation):
- depthContribution = (depthScore/5) * 60
- bloomContribution = (bloomLevel/5) * 40
- conceptScore = depthContribution + bloomContribution (out of 100)
- baseScore = average conceptScore across all concepts
- If misconception caught: add 8 points (max 100)
- If misconception missed: subtract 5 points (min 0)
- Round to integer

SECTION 12 — AI MODEL ASSIGNMENTS
--------------------------------------------------------------------------------

Groq (primary provider):
API base: https://api.groq.com/openai/v1
SDK: npm install groq-sdk
Auth: GROQ_API_KEY environment variable
Free tier: ~14,400 requests/day per model

Gemini (secondary provider):
API base: https://generativelanguage.googleapis.com/v1beta
No SDK required (use fetch directly)
Auth: GEMINI_API_KEY environment variable
Free tier: 1,500 requests/day on Flash

Model Assignment Table:

Feature 1 (concept tree + misconception):
  Provider: Gemini
  Model: gemini-2.0-flash
  Reason: responseMimeType JSON enforcement, reliable structured output
  Temperature: 0.7
  Max tokens: 1000

Feature 2 (Mia conversation streaming):
  Provider: Groq
  Model: llama-3.3-70b-versatile
  Reason: 500-800 tokens/second streaming speed, good persona following
  Temperature: 0.7
  Max tokens: 500

Feature 6 (misconception evaluator, optional):
  Provider: Groq
  Model: llama-3.1-8b-instant
  Reason: Fastest and cheapest, binary classification task
  Temperature: 0.0
  Max tokens: 20

Feature 7 (report generation):
  Provider: Gemini
  Model: gemini-2.0-flash
  Reason: Most complex structured JSON output, responseMimeType enforcement
  Temperature: 0.4
  Max tokens: 1000

Rate Limit Fallback:
- If Groq returns 429: wait 3 seconds, retry once
- If still failing: fall back to gemini-2.0-flash for that call
- Never surface the error to the user

Gemini API Call Pattern:
fetch("https://generativelanguage.googleapis.com/v1beta/models/gemini-2.0-flash:generateContent?key=" + GEMINI_KEY, {
  method: "POST",
  headers: { "Content-Type": "application/json" },
  body: JSON.stringify({
    contents: [{ parts: [{ text: prompt }] }],
    generationConfig: {
      responseMimeType: "application/json",
      temperature: 0.4
    }
  })
})

Groq Streaming Pattern:
import Groq from "groq-sdk";
const groq = new Groq({ apiKey: process.env.GROQ_API_KEY });
const stream = await groq.chat.completions.create({
  model: "llama-3.3-70b-versatile",
  messages: [...],
  max_tokens: 500,
  stream: true,
  temperature: 0.7
});
for await (const chunk of stream) {
  const token = chunk.choices[0]?.delta?.content || "";
  res.write(token);
}

SECTION 15 — CRITICAL RULES (NEVER VIOLATE)
--------------------------------------------------------------------------------

1. Mia never explains, teaches, or gives information. She only reacts, asks,
   and confirms. If she starts explaining, the system prompt needs fixing.

2. Metadata must never appear in the visible chat. stripPartialMetadata must
   run on every streaming chunk, not just at the end.

3. Conversation history passed to the API must always use clean_text
   (metadata stripped). Passing raw responses with metadata breaks Mia's
   persona by making her aware of the scoring system.

4. The user never learns misconception detection is running during the session.
   No UI element, no indicator, nothing. Silent.

5. The report is stored permanently before rendering. Never regenerate it on
   revisit. Always serve from database.

6. Session state is saved every 5 messages minimum. A crash should never wipe
   more than 5 messages.

7. The shareable card download must work. This is the primary viral and demo
   sharing mechanism. Test it on multiple browsers.

8. Rate limit errors from Groq must never surface to the user. Retry once,
   then fall back to Gemini silently.

9. All depth score updates use max(existing, new). Depth scores never go
   backward. If the user explains something more shallowly later, the highest
   score for that concept is what counts.

10. The report opening summary must accurately reflect the actual depth scores.
    If it praises something the scores show as shallow, trust in the product
    collapses. Validate this before demo.
